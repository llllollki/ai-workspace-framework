# 0001 — Data Boundary: alh-tracker vs. AssistedLivingHelp

**Date:** 2026-05-05
**Status:** accepted
**Supersedes:** n/a
**Superseded by:** n/a

## Context

alh-tracker and AssistedLivingHelp share a commercial relationship: some RCFE facilities are both ALH listing partners and alh-tracker customers. This creates a surface where the two products could share data. The risk is that resident care data — shift logs, care observations, incident records, medication observations — could inadvertently flow into the ALH placement and lead-matching platform, exposing sensitive care information to a commercial system the resident never consented to.

The products serve fundamentally different purposes: ALH connects families with facilities; alh-tracker logs care operations inside a facility. The data generated by each product is appropriate for its context and inappropriate for the other.

## Decision

Resident care data in alh-tracker must not flow to the AssistedLivingHelp platform under any circumstance without all of the following: explicit named operator consent describing the specific data categories and purpose, legal and compliance counsel review, a documented data sharing agreement, technical enforcement at the query and API layer, and an audit trail of what was shared and under what consent basis.

The only data that crosses the boundary as part of normal commercial operations is the commercial relationship status (`alh_partner` boolean and `alh_partner_tier` field on the Facility entity). This communicates that a facility is an ALH partner — nothing more.

Data that must remain siloed in alh-tracker: resident identity, shift log entries, care log records, handoff summaries, observed care task records, incident logs, follow-up records, caregiver accounts, and caregiver activity logs.

This boundary is an architectural constraint, not a policy preference. No ALH platform query path should ever reach care log data. No shared API endpoint should return care log data to an ALH-authenticated request.

## Consequences

**Easier:** Resident privacy is structurally protected. Regulatory exposure from inadvertent data sharing is prevented. Facility trust is preserved — operators can rely on alh-tracker not leaking care data to the platform that manages their business relationships. Future cross-product features requiring data sharing have a clear, enforceable consent and legal process to follow.

**Harder:** Cross-product features that surface care data in an ALH context (e.g., a future "operator health score" derived from shift log quality) require significant legal and technical work before they can be built. The data boundary must be verified in code review and security review for any feature touching both products.
